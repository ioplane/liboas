# JSON Schema 2020-12

liboas реализует валидацию JSON Schema 2020-12, как того требует OpenAPI 3.2.
Представление схемы определено в `oas_schema.h`.

## Структура oas_schema_t

`oas_schema_t` -- плоская структура, содержащая все возможные ключевые слова
JSON Schema. Неиспользуемые поля содержат значения-сентинели (`-1` для
опциональных целых чисел, `nullptr` для указателей, `false` для булевых).
Структура аллоцируется через арену вызовом `oas_schema_create()`.

## Система битовых масок типов

Типы представлены битовой маской в `type_mask`, что позволяет схемам принимать
несколько типов (напр. `["string", "null"]`).

| Константа         | Значение | Тип JSON Schema |
|-------------------|----------|-----------------|
| `OAS_TYPE_NULL`   | `0x01`   | `null`          |
| `OAS_TYPE_BOOLEAN`| `0x02`   | `boolean`       |
| `OAS_TYPE_INTEGER`| `0x04`   | `integer`       |
| `OAS_TYPE_NUMBER` | `0x08`   | `number`        |
| `OAS_TYPE_STRING` | `0x10`   | `string`        |
| `OAS_TYPE_ARRAY`  | `0x20`   | `array`         |
| `OAS_TYPE_OBJECT` | `0x40`   | `object`        |

`type_mask` равный `0x00` означает отсутствие ограничения по типу (допускается
любой тип). Используйте `oas_type_from_string()` для преобразования имени типа
в значение битовой маски.

Согласно JSON Schema, `integer` является подмножеством `number`. Когда установлен
бит `OAS_TYPE_NUMBER`, целочисленные значения также проходят валидацию.

## Ограничения по типам

### Строковые ограничения

| Поле         | Тип           | Описание                                 |
|--------------|---------------|------------------------------------------|
| `min_length` | `int64_t`     | Минимальная длина строки (-1 = не задано) |
| `max_length` | `int64_t`     | Максимальная длина строки (-1 = не задано)|
| `pattern`    | `const char*` | Регулярное выражение ECMA-262 (валидируется regex-бэкендом) |
| `format`     | `const char*` | Подсказка формата (напр. `"email"`, `"uuid"`) |

Сопоставление с образцом использует vtable regex-бэкенда, по умолчанию --
QuickJS libregexp для соответствия ECMA-262. Образцы не привязаны к якорям
согласно спецификации JSON Schema.

### Числовые ограничения

| Поле                 | Тип      | Описание                           |
|----------------------|----------|------------------------------------|
| `minimum`            | `double` | Минимальное значение (включительно)|
| `maximum`            | `double` | Максимальное значение (включительно)|
| `exclusive_minimum`  | `double` | Исключающий минимум                |
| `exclusive_maximum`  | `double` | Исключающий максимум               |
| `multiple_of`        | `double` | Значение должно делиться на это    |

Каждое числовое поле имеет соответствующий булев флаг `has_*`, поскольку 0.0
является допустимым значением ограничения и не может служить сентинелем.

### Ограничения массивов

| Поле                | Тип              | Описание                            |
|---------------------|------------------|-------------------------------------|
| `items`             | `oas_schema_t *` | Схема для всех элементов массива    |
| `prefix_items`      | `oas_schema_t **`| Валидация кортежа (позиционные схемы)|
| `prefix_items_count`| `size_t`         | Количество схем prefix item         |
| `min_items`         | `int64_t`        | Минимальная длина массива (-1 = не задано) |
| `max_items`         | `int64_t`        | Максимальная длина массива (-1 = не задано)|
| `unique_items`      | `bool`           | Все элементы должны быть уникальны  |
| `contains`          | `oas_schema_t *` | Хотя бы один элемент должен соответствовать |
| `min_contains`      | `int64_t`        | Минимум соответствующих элементов для contains |
| `max_contains`      | `int64_t`        | Максимум соответствующих элементов для contains |

Когда задан `prefix_items`, элементы по каждому индексу валидируются по
соответствующей схеме. Элементы за пределами префикса валидируются по `items`.

### Ограничения объектов

| Поле                       | Тип                         | Описание                          |
|----------------------------|-----------------------------|-----------------------------------|
| `properties`               | `oas_property_t *`          | Связный список именованных свойств|
| `properties_count`         | `size_t`                    | Количество свойств                |
| `required`                 | `const char **`             | Имена обязательных свойств        |
| `required_count`           | `size_t`                    | Количество обязательных свойств   |
| `additional_properties`    | `oas_schema_t *`            | Схема для дополнительных свойств  |
| `additional_properties_bool` | `bool`                    | True если задано как булево false  |
| `min_properties`           | `int64_t`                   | Минимальное количество свойств    |
| `max_properties`           | `int64_t`                   | Максимальное количество свойств   |
| `property_names`           | `oas_schema_t *`            | Схема, которой должны соответствовать все имена свойств |
| `pattern_properties`       | `oas_pattern_property_t *`  | Схемы свойств по regex-ключу      |
| `dependent_required`       | `oas_dependent_required_t *`| Условные обязательные свойства    |
| `dependent_schemas`        | `oas_dependent_schema_t *`  | Условное применение схем          |

Свойства хранятся в виде связного списка (`oas_property_t`). Каждый узел имеет
поля `name`, `schema` и указатель `next`. Для добавления используйте
`oas_schema_add_property()`.

## Ключевые слова композиции

| Поле        | Тип              | Ключевое слово JSON Schema |
|-------------|------------------|----------------------------|
| `all_of`    | `oas_schema_t **`| `allOf` -- все должны совпасть |
| `any_of`    | `oas_schema_t **`| `anyOf` -- хотя бы одна должна совпасть |
| `one_of`    | `oas_schema_t **`| `oneOf` -- ровно одна должна совпасть |
| `not_schema`| `oas_schema_t *` | `not` -- не должна совпасть |

Каждый массив композиции имеет соответствующее поле `*_count`.

## Условные ключевые слова

| Поле          | Тип              | Ключевое слово JSON Schema |
|---------------|------------------|----------------------------|
| `if_schema`   | `oas_schema_t *` | `if`                       |
| `then_schema` | `oas_schema_t *` | `then`                     |
| `else_schema` | `oas_schema_t *` | `else`                     |

Когда `if` проходит валидацию успешно, применяется `then`. В противном случае
применяется `else`. `then` и `else` не действуют без `if`.

## Дискриминатор

Полиморфизм OpenAPI через дискриминатор:

- `discriminator_property` -- имя свойства для выбора схемы
- `discriminator_mapping` -- массив соответствий ключ-`$ref`
- `discriminator_mapping_count` -- количество записей соответствий

Значение свойства дискриминатора определяет, какая подсхема `oneOf`/`anyOf`
используется для валидации, избавляя от необходимости перебирать все варианты.

## Const и Enum

- `const_value` (`yyjson_val *`) -- значение должно быть точно равно (глубокое сравнение).
- `enum_values` (`yyjson_val *`) -- значение должно быть одним из перечисленных.

Оба ссылаются на zero-copy значения yyjson из разобранного документа.

## Разрешение $ref

- `ref` -- исходная строка `$ref` (напр. `"#/components/schemas/Pet"`).
- `ref_resolved` -- указатель на разрешённую целевую схему после разрешения.

Когда `ref_resolved` установлен, валидатор следует по нему прозрачно. Исходная
строка `ref` сохраняется для диагностики и эмиссии.

## Расширенные ключевые слова 2020-12

| Поле                        | Тип              | Описание                           |
|-----------------------------|------------------|------------------------------------|
| `unevaluated_properties`    | `oas_schema_t *` | Схема для свойств, не покрытых `properties`, `patternProperties` или композицией |
| `unevaluated_items`         | `oas_schema_t *` | Схема для элементов массива, не покрытых `prefixItems` или `items` |

Эти ключевые слова требуют отслеживания того, какие свойства/элементы были
«вычислены» при валидации ключевых слов композиции.

## Контекст чтения/записи

- `read_only` -- свойство присутствует только в ответах.
- `write_only` -- свойство присутствует только в запросах.
- `deprecated` -- помечает схему как устаревшую.
- `nullable` -- флаг совместимости с OAS 3.0; внутренне добавляет `OAS_TYPE_NULL` в `type_mask`.
