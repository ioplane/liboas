# Обзор архитектуры

liboas -- библиотека на C23 для парсинга, валидации и обслуживания спецификаций OpenAPI 3.2.
Целевая платформа -- исключительно Linux (ядро 6.7+, glibc 2.39+), лицензия GPLv3.

## Двухслойная архитектура

```
                        OpenAPI 3.2 JSON/YAML
                               |
                     +---------+---------+
                     |   Layer 1: Parse  |
                     |   (OAS Model)     |
                     +---------+---------+
                               |
                          oas_doc_t
                               |
                     +---------+---------+
                     |  Layer 2: Compile |
                     |  & Validate       |
                     +---------+---------+
                               |
               +---------------+---------------+
               |               |               |
         Validate Req    Validate Resp    Emit JSON
```

### Слой 1 -- модель OAS (этап парсинга)

Преобразует документ OpenAPI 3.2 (JSON или YAML) в дерево C-структур в памяти.
Все аллокации проходят через единый арена-аллокатор.

| Компонент         | Заголовок         | Роль                                     |
|-------------------|-------------------|------------------------------------------|
| Парсер            | `oas_parser.h`    | JSON-строка/файл в `oas_doc_t`           |
| Модель документа  | `oas_doc.h`       | Структуры для путей, операций, ответов   |
| Модель схемы      | `oas_schema.h`    | Представление JSON Schema 2020-12        |
| Резолвер `$ref`   | (внутренний)      | Разрешение ссылок с обнаружением циклов  |
| Билдер            | `oas_builder.h`   | Программное построение документа         |
| Эмиттер           | `oas_emitter.h`   | `oas_doc_t` обратно в JSON-строку        |

### Слой 2 -- скомпилированная среда выполнения (этап валидации)

Прекомпилирует деревья схем в плоские массивы инструкций, затем валидирует
HTTP-запросы и ответы по скомпилированному представлению.

| Компонент            | Заголовок         | Роль                                      |
|----------------------|-------------------|-------------------------------------------|
| Компилятор схем      | `oas_compiler.h`  | Дерево схем в инструкционный байткод       |
| Regex-бэкенд         | `oas_regex.h`     | Сопоставление с образцом ECMA-262 (vtable) |
| ВМ валидации         | `oas_validator.h` | Выполнение байткода на JSON-значениях      |
| Матчер путей         | (внутренний)      | Сопоставление шаблонов с извлечением параметров |
| Валидатор запросов   | `oas_validator.h` | Полная валидация HTTP-запросов             |
| Валидатор ответов    | `oas_validator.h` | Полная валидация HTTP-ответов             |
| Адаптер              | `oas_adapter.h`   | Единый фасад для интеграции middleware     |

## Модель памяти

### Арена-аллокатор

Все узлы модели OAS аллоцируются из единой `oas_arena_t`. Когда документ
больше не нужен, один вызов `oas_arena_destroy()` освобождает всё за O(1).

```c
oas_arena_t *arena = oas_arena_create(0);   /* 64 KiB default blocks */
oas_doc_t *doc = oas_doc_parse(arena, json, len, errors);
/* ... use document ... */
oas_arena_destroy(arena);                    /* frees all model nodes */
```

Арена растёт, выделяя новые блоки по мере необходимости. `oas_arena_reset()`
возвращает всю память без освобождения нижележащих блоков, что удобно для повторного использования.

### Zero-copy строки

Строковые поля в `oas_doc_t` и `oas_schema_t` указывают непосредственно в
буфер документа yyjson. Документ yyjson (`doc->_json_doc`) должен оставаться
живым, пока на эти строки есть ссылки. Вызовите `oas_doc_free()` для
освобождения JSON-документа, когда строковые данные больше не нужны.

## Структура каталогов

```
include/liboas/     Публичные заголовки API
src/core/           Арена-аллокатор, список ошибок, интернирование строк
src/parser/         Парсинг JSON/YAML, разрешение $ref, JSON Pointer
src/model/          Построение модели документа
src/compiler/       Компиляция схем в байткод
src/validator/      ВМ валидации, валидация запросов/ответов
src/emitter/        Эмиссия JSON
src/adapter/        Адаптер интеграции с iohttp
vendor/libregexp/   Vendored QuickJS libregexp (регулярные выражения ECMA-262)
tests/unit/         Модульные тесты на Unity
tests/fuzz/         Фаззинг-цели LibFuzzer
```

## Граф зависимостей компонентов

```
oas_adapter
    |
    +-- oas_parser --> oas_doc (model)
    |                     |
    |                     +-- oas_schema
    |                     +-- $ref resolver
    |
    +-- oas_compiler --> oas_compiled_doc
    |       |
    |       +-- oas_regex (backend vtable)
    |
    +-- oas_validator
    |       |
    |       +-- path matcher
    |       +-- content negotiation (oas_negotiate)
    |
    +-- oas_emitter
    +-- oas_problem (RFC 9457)
```

## Принципы проектирования

- **Чистый C23**: никаких зависимостей от C++, обязателен флаг `-std=c23`.
- **Арена в первую очередь**: единая стратегия аллокации на весь жизненный цикл модели.
- **Zero-copy**: строковые указатели в исходный JSON-буфер.
- **Компиляция, затем валидация**: схемы прекомпилируются один раз, валидируются многократно.
- **Абстракция через vtable**: regex-бэкенд подключаемый через указатели на функции.
- **Отрицательные errno при возврате**: все функции, допускающие ошибку, возвращают 0 при успехе
  или `-ENOMEM`, `-EINVAL`, `-ENOENT` при ошибке.
- **Только Linux**: без бремени переносимости на Windows или macOS.
