## Sprint 17: Remote $ref (File + HTTP/HTTPS)

### Problem

`oas_ref_resolve_all()` only resolves internal `#/components/schemas/X` references. Real-world specs use external refs:

```yaml
components:
  schemas:
    Error:
      $ref: "https://registry.company.com/schemas/error.yaml#/Error"
    Pet:
      $ref: "./schemas/pet.yaml#/Pet"
```

Without external $ref support, liboas cannot parse most multi-file or shared-schema specifications.

### Architecture

**Layered resolution strategy — 4 levels:**

| Level | Condition | Implementation |
|-------|-----------|----------------|
| File refs | Always available | `fopen`/`fread`, base_dir relative |
| HTTP refs | `allow_remote=true` | Built-in POSIX sockets (~150 LOC) |
| HTTPS refs | User callback | Any TLS stack (wolfSSL, OpenSSL, iohttp, libcurl) |
| HTTPS refs | `-DOAS_TLS=ON` | Built-in wolfSSL or OpenSSL fallback |

**Resolution priority:**
1. User-provided `fetch` callback (if set) — handles ALL remote URLs
2. Built-in TLS fetcher (if compiled with `-DOAS_TLS=ON`) — HTTPS
3. Built-in POSIX sockets — HTTP only
4. `-ENOTSUP` with descriptive error for HTTPS without TLS

### Components

#### URI Parser (`src/parser/oas_uri.c`)
- RFC 3986 parsing: scheme, authority, host, port, path, query, fragment
- Relative URI resolution against base URI
- Path traversal prevention (no `../../etc/passwd`)

#### Document Cache (`src/parser/oas_ref_cache.c`)
- URI → parsed document map
- Deduplication: same URI fetched once
- Configurable max documents (default: 100)
- Owns yyjson_doc lifetime for cached documents

#### File Loader (`src/parser/oas_ref_file.c`)
- Read JSON/YAML files relative to base_dir
- base_dir defaults to dirname of source document
- Supports `.json` and `.yaml`/`.yml` (when OAS_YAML=ON)
- Max file size enforcement

#### HTTP Fetcher (`src/parser/oas_ref_fetch.c`)
- Built-in: POSIX sockets, HTTP/1.1 GET, Content-Length parsing
- Redirect following (max configurable, default 5)
- Timeout via `setsockopt(SO_RCVTIMEO)`
- Max response size enforcement

#### TLS Backend (optional, `src/parser/oas_tls_wolfssl.c` / `oas_tls_openssl.c`)
- Compiled only with `-DOAS_TLS=ON`
- CMake option `OAS_TLS_BACKEND` selects wolfssl (default) or openssl
- Wraps socket fd with TLS read/write

#### Ref Resolver Extension (`src/parser/oas_ref.c`)
- Integrate file loader + HTTP fetcher into existing resolve pipeline
- Cross-document cycle detection (URI + JSON Pointer pair tracking)
- Recursive resolution: remote doc may contain further $ref

### Public API

```c
/* User-provided fetch callback */
typedef int (*oas_ref_fetch_fn)(void *ctx, const char *url,
                                 char **out_data, size_t *out_len);

typedef struct {
    bool allow_remote;          /* HTTP/HTTPS (default: false) */
    bool allow_file;            /* file refs (default: true) */
    oas_ref_fetch_fn fetch;     /* user override (priority over built-in) */
    void *fetch_ctx;            /* user context */
    const char *base_dir;       /* for file refs (default: dirname of source) */
    int max_documents;          /* cache limit (default: 100) */
    int fetch_timeout_ms;       /* built-in fallback (default: 30000) */
    size_t max_fetch_size;      /* built-in fallback (default: 10 MB) */
    int max_redirects;          /* built-in fallback (default: 5) */
} oas_ref_options_t;
```

Passed via extended parse API:
```c
oas_doc_t *oas_doc_parse_ex(oas_arena_t *arena, const char *json, size_t len,
                             const oas_ref_options_t *ref_opts,
                             oas_error_list_t *errors);
```

## Sprint 18: Real-world Validation Hardening

### 18.1: readOnly/writeOnly Enforcement

**Problem:** `read_only` and `write_only` are parsed but not enforced. readOnly fields should be rejected in request bodies, writeOnly in response bodies.

**Implementation:**
- New opcodes: `OAS_OP_CHECK_READ_ONLY`, `OAS_OP_CHECK_WRITE_ONLY`
- Compiler emits them when schema has `read_only=true` or `write_only=true`
- Validator needs direction context: `OAS_DIR_REQUEST` or `OAS_DIR_RESPONSE`
- `oas_validate_request()` sets REQUEST, `oas_validate_response()` sets RESPONSE
- `oas_validate()` (standalone) ignores read/writeOnly (no direction context)

**Tests (6):** readOnly_rejected_in_request, readOnly_allowed_in_response, writeOnly_rejected_in_response, writeOnly_allowed_in_request, nested_readOnly_in_allOf, readOnly_ignored_standalone_validate

### 18.2: Default Value Application

**Problem:** Schema `default` values are parsed but not applied during validation. Missing fields with defaults should be filled in.

**Implementation:**
- New function: `oas_validate_request_with_defaults()` — validates AND mutates
- Uses yyjson mutable doc: copies input, applies defaults, validates
- Recursive: walks properties, items, nested objects
- Only for request validation (response defaults don't make sense)
- Does NOT overwrite existing values

```c
int oas_validate_request_with_defaults(const oas_compiled_doc_t *doc,
                                        oas_http_request_t *req,
                                        oas_validation_result_t *result,
                                        oas_arena_t *arena,
                                        char **out_body, size_t *out_len);
```

Returns modified body JSON in `out_body` (caller frees with `free()`).

**Tests (5):** default_applied_missing_field, default_not_overwrite_existing, default_nested_object, default_array_items, default_no_mutation_on_validation_fail

### 18.3: Cookie Parameter Validation

**Problem:** Cookie parameters are defined in OAS specs (`in: "cookie"`) but not validated in `oas_validate_request()`.

**Implementation:**
- Add `const char *cookie_header` field to `oas_http_request_t`
- Cookie parser: `name=value; name2=value2` → name/value pairs
- Percent-decoding of cookie values
- Validate against operation parameters where `in == "cookie"`
- Required cookie check

**Tests (5):** cookie_parse_single, cookie_parse_multiple, cookie_validate_required_missing, cookie_validate_schema, cookie_percent_decode

## Key Design Decisions

1. **Fetch callback pattern** — liboas is server-agnostic, user provides HTTP client from their stack (iohttp, h2o, mongoose, libcurl, etc.)
2. **Built-in HTTP fallback** — POSIX sockets for standalone use (CLI tools, CI, codegen)
3. **Optional TLS** — `-DOAS_TLS=ON` with wolfSSL (default) or OpenSSL, for standalone HTTPS without user callback
4. **No new mandatory dependencies** — TLS is opt-in at compile time
5. **Document cache** — fetched documents cached by URI, configurable limit
6. **Direction-aware validation** — readOnly/writeOnly require request/response context
